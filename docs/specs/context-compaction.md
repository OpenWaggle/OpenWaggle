# Spec: Context Compaction for Long Conversations

**Status:** Draft / Notes
**Priority:** Medium
**Discovered during:** QA test G (restart-continuation after orchestration)

## Problem

When a conversation grows large (e.g., after an orchestration run with 5 sub-agents), the full message history is sent to the API on every new message. This can exceed the model's context window, causing the API to return empty responses or fail silently.

### Observed Behavior

- After an orchestration run (test F), restarting the app and continuing the thread (test G) produced two messages with **zero text output**.
- Server logs showed `stream-processing: 0ms`, `toolCalls: 0` — the model returned immediately with nothing.
- Root cause: `conversationToMessages()` converts ALL persisted messages into the API payload. After orchestration, this payload is very large (5 sub-agent results, many tool-call/tool-result pairs).

### Current Message Flow (no truncation)

```
Persisted messages (all)
  -> conversationToMessages()     [message-mapper.ts]
  -> [...existingMessages, newUserMessage]
  -> chat()                       [TanStack AI]
  -> API call with full history
```

## Requirements

### Must Have
- Conversations must remain functional regardless of history length.
- Compaction must preserve enough context for the model to produce coherent responses.
- The mechanism must work across app restarts (compaction state must be persisted or reconstructable).

### Should Have
- Per-provider context window awareness — different models have different limits (e.g., Claude 200K, GPT-4o 128K, Gemini 1M+).
- Graceful degradation — if compaction fails, the conversation should still work (even if with a truncated or degraded history).
- Tool-call/tool-result pairing integrity — compacted history must never break the tool_use/tool_result pairing required by the Anthropic API.

### Nice to Have
- User visibility into when compaction occurs (e.g., a subtle indicator).
- Ability to "expand" compacted history for debugging.

## Design Considerations

### 1. Provider Context Window Registry

Each provider/model needs a known input token limit. This doesn't exist today — `ProviderDefinition` only has `maxTokens` (output) and sampling config.

```typescript
interface ModelCapabilities {
  contextWindow: number      // total input + output tokens
  maxOutputTokens: number    // current maxTokens
}
```

Need to decide:
- Hardcoded per known model? Or fetched from provider APIs?
- How to handle custom/OpenRouter models with unknown limits?
- Safety margin (e.g., reserve 20% for system prompt + tools + output)?

### 2. Token Counting

To know when to compact, we need to estimate token count of the message history. Options:

- **Rough estimate**: character count / 4 (fast, inaccurate)
- **Tiktoken**: Accurate for OpenAI models, approximate for others
- **Provider-specific tokenizers**: Most accurate but complex
- **API-reported usage**: Use `usage.input_tokens` from previous responses to calibrate

### 3. Compaction Strategies

Several approaches, not mutually exclusive:

#### A. Sliding Window with Summary
- Keep last N messages verbatim.
- Summarize older messages into a single "conversation summary" message.
- Summary generated by the model itself (or a cheaper model).
- Tradeoff: requires an extra API call for summarization.

#### B. Smart Truncation
- Drop oldest messages until under the token limit.
- Always keep: system prompt, first user message (sets context), last K messages.
- Drop tool-call/tool-result pairs as units (never break pairing).
- Tradeoff: loses context without any summary.

#### C. Selective Compaction
- Keep all user messages (short).
- Summarize/truncate long assistant messages (especially tool results).
- Tool results are often the largest parts (webFetch output, file contents).
- Tradeoff: more complex logic, better context preservation.

#### D. Tiered Approach
- Recent messages: full fidelity.
- Medium-age messages: summarized tool results, full text.
- Old messages: single-line summaries.
- Tradeoff: most complex but best quality.

### 4. When to Compact

- **Before sending**: Check estimated token count before each `chat()` call. If over limit, compact.
- **After receiving**: Use `usage.input_tokens` from the response to calibrate estimates.
- **On persist**: Compact at save time so restarts don't need to re-compact (but this changes the persisted data).

Important: compaction should happen in the message-preparation layer (`agent-loop.ts`), NOT modify the persisted conversation. The full history should always be preserved on disk.

### 5. Orchestration-Specific Considerations

Orchestration produces particularly large histories because:
- Sub-agent results can be very long (full analysis reports).
- Multiple tool-call/tool-result pairs per sub-agent.
- The synthesis message references all sub-agent outputs.

Orchestration results might benefit from a dedicated compaction strategy:
- Replace detailed sub-agent results with their summaries.
- Keep the synthesis message in full.

### 6. Message-Mapper Considerations

`conversationToMessages()` currently:
- Filters tool-calls to only include those with matching tool-results (good).
- Converts all text parts verbatim (no truncation).
- Includes all tool-result content verbatim (can be very large).

This is where compaction logic would plug in — either as a wrapper around `conversationToMessages()` or as a post-processing step.

## Open Questions

1. Should we count tokens before or after adding the system prompt and tool definitions? (System prompt + tools can be significant.)
2. Should compaction be visible to the user? (e.g., "Context compacted — older messages summarized")
3. Should we support a "model-driven summary" approach where we ask the model to summarize before compacting? This produces better summaries but costs an extra API call.
4. How do we handle the case where even the system prompt + tools + last message exceeds the context window?
5. Should compacted state be cached so we don't re-compute on every message?
6. How does this interact with waggle mode (which has its own message structure)?

## Implementation Order (Suggested)

1. **Add context window limits to provider/model registry** — prerequisite for everything.
2. **Add token estimation** — rough character-based first, refine later.
3. **Implement smart truncation (Strategy B)** — simplest, handles the crash case.
4. **Add sliding window with summary (Strategy A)** — better UX, requires summary generation.
5. **Orchestration-specific compaction** — optimize for the most common large-context case.
6. **Per-provider tuning** — different strategies/thresholds per provider.
