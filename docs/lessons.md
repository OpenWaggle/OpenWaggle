# Lessons

User corrections and behavioral rules. Updated whenever the user corrects the agent. These are patterns to never repeat.

## Active Rules

- Do not add another repository’s library-development notes to OpenWaggle knowledge files. `docs/learnings.md` and `docs/lessons.md` should only record information that affects OpenWaggle itself, except when an external library is being integrated into OpenWaggle.
- Do not treat urgency language like “ASAP” as approval to commit or push. If the user has not explicitly approved a commit, leave fixes local for review.
- When an E2E suite fails at app startup or most tests fail with the same missing-element symptom, inspect the rendered error screen/screenshot and runtime console first; do not continue treating failures as stale assertions until the app-level crash is resolved.
- For release/versioning design grilling, do not infer that the user dislikes alpha/beta terminology from criticism of the current release setup; distinguish wording preferences from process/channel/versioning-model objections.
- For OpenWaggle release/update UX, treat in-app update notification and Settings-based alpha/beta opt-in/out as first-class product requirements from the beginning; do not push prerelease users toward scripts, manual downloads, or side-install workflows unless explicitly requested.
- In release-channel grilling, once Stable/Beta/Alpha risk boundaries are defined, do not ask whether lower-risk tracks should receive higher-risk builds; strict containment is implied unless the user explicitly proposes otherwise.
- When the user wants one large migration MR/PR, do not create or switch to side branches for adjacent docs, skills, or planning work; keep changes on the single active migration branch unless explicitly told otherwise.
- Before deleting migration/planning/spec documents, reconcile their planned work against the current implementation and extract any remaining actionable gaps into a replacement spec; never assume a historical-looking spec is fully implemented.
- Keep the `/release` skill and release/versioning policy runtime- and SDK-agnostic; do not bake Pi-specific terminology or assumptions into release skills because OpenWaggle may support other CLIs/SDKs in the future.
- Do not recreate a skill location that the user intentionally removed; when the user removes a `.codex`, `.claude`, `.agents`, or `.openwaggle` skill copy, preserve that location choice unless explicitly asked to restore it.
- For any OpenWaggle implementation modeled after T3Code, inspect the actual T3Code source before judging correctness or approach quality; compare against its real implementation, not assumptions or summaries, including the lower-level primitives it composes.
- For Pi migration design grilling, first ask what Pi TUI/SDK already does and treat that as the default answer. Do not ask the user to confirm obvious Pi-parity behavior; only ask where OpenWaggle must intentionally diverge or add desktop/product semantics beyond Pi TUI. Error retry behavior must mirror Pi's native auto-retry semantics rather than inventing an OpenWaggle-specific retry model. Branch-summary choice must mirror Pi's tree-navigation timing: ask when the draft branch is created/non-head node is selected, not later when the user sends. If the user chooses to summarize at draft creation, that explicit action can materialize the draft into a proper branch even before a new user message is sent.
- For compaction UX grilling specifically, do not ask questions whose answer is simply “yes, follow Pi TUI behavior.” Treat Pi TUI behavior as locked unless OpenWaggle needs an explicit desktop/UI-layer divergence.
- For session tree UX, do not show the full Pi node tree in the left project/session sidebar. The left sidebar is navigation-first: show sessions, materialized branches that have actual content/work, and transient draft branch rows as immediate visual feedback when the user selects/branches from a non-head node. Clicking a materialized branch in the left sidebar always navigates to that branch's current head/full conversation. Draft rows must update or disappear if the user moves on/clicks around without sending, and become durable branch navigation only after the next send materializes the Pi branch. Expose the full Pi session tree through an explicit right-side Session Tree panel opened from the header, matching Pi TUI's `/tree` as an on-demand navigation surface, ship the full Pi tree filter set (Default, No tools, User only, Labeled, All), and implement its keyboard shortcuts with `@tanstack/hotkeys`. In the chat transcript, show the Git branch-out affordance everywhere the user can validly branch out, not only in the Session Tree panel. OpenWaggle may delete/archive full sessions, but branch lifecycle is archive/restore only until Pi supports native branch deletion; archived branches are hidden from normal left-sidebar navigation, remain visible in the full Session Tree with archived state, and are managed from the Settings archived-items/messages section. Branch row actions are opened from a hover-revealed three-dots overflow button, and branch rename is inline in the sidebar row. Pi-style clone-to-new-session is session/current-position scoped and belongs in the left-sidebar session overflow menu. Pi-style fork-to-new-session is previous-user-message scoped and belongs on user message row actions next to copy/other transcript actions. Support both `/clone` and `/fork` slash parity and include both actions in the command palette; command palette actions may open the needed selector/context flow rather than being the primary visual affordance.
- During migration grilling, do not re-ask decisions already agreed in the plan or current conversation unless the user raises a conflicting idea. Treat agreed decisions as locked constraints and move to unresolved implementation/product choices only.
- New renderer/global keyboard shortcuts must use `@tanstack/hotkeys`; do not add more ad hoc `window`/`document` keydown listeners. Migration specs that add hotkeys should include a repository pass to identify existing shortcut handlers and consolidate suitable app-level shortcuts onto TanStack Hotkeys while leaving component-local text-editor commands inside their owning editor systems when appropriate.
- When another agent is concurrently doing migration work, assume their work may not include the latest decisions from this conversation. Avoid overlapping edits where possible, inspect staged/dirty files before touching anything, and treat their changes as inputs to improve/reconcile rather than overwrite.
- When the user reports perceived UI latency, do not rely only on route/ARIA E2E timing. Inspect the actual layout and animation model for first visual response, especially whether heavy content participates in layout during transitions.
- When the user disables MCP because it bloats context, do not call MCP again in that run. Use code inspection and non-MCP checks instead, and clearly report any live-QA gap.
- In grill-me mode, stop once the core preference is clear; consolidate the decision instead of re-asking the same positioning question through small variations.
- Manual setup guides must name exact UI selections and defaults, not only high-level recommendations. If a user is looking at a settings screen, provide checkbox-by-checkbox guidance for every visible option/sub-option and explain when an option should be deferred until CI/status checks exist.
- In grill-me mode, when the user has already set a global design rule (for example infer every type as deeply as possible), do not ask obvious derivative questions as if they are important unresolved decisions; infer and lock the obvious consequence, then ask only about real trade-offs or edge cases.
- Avoid broad tool reads that dump large documents into context after compaction. Use the smallest targeted reads/edits necessary and prefer existing handoff context when it is sufficient.
- When asked to summarize an app/codebase, inspect the actual implementation files first instead of relying only on README or architecture docs; explicitly ground the summary in code-level evidence.
- Keep strategic design responses tight when the user asks a focused follow-up; do not bury the core recommendation under long explanatory text if the user is actively iterating on product semantics.
- Prefer the simplest first product interaction model during architecture grilling; do not introduce slash-command-based configuration flows unless the user explicitly wants that level of flexibility in v1.
- When the user wants to stay on the main architecture/spec thread, do not shift the conversation into a side spec or artifact as the new focus without explicit confirmation. Supporting docs are fine, but keep the primary spec thread primary.
- When the user explicitly narrows scope to high-impact migration topics, do not spend grill-me turns re-asking about areas they already said should remain mostly unchanged. Use the stated stable areas as constraints and ask only architecture-critical questions.
- For Waggle migration planning, treat “everything behaves like standard mode unless the user explicitly wants a difference” as the default constraint. Waggle should be implemented as Pi extension/in-session behavior rather than separate runtimes, so standard-mode branch navigation, draft creation, materialization, archive/restore, transcript branch-out, and active-run semantics apply equally to Waggle. Waggle's differences are its independent branch-scoped run configuration and transcript UX coloring/attribution. Do not ask repeated same-behavior questions for composer, settings, diffs, git, or adjacent integrations unless there is a serious architecture conflict.
- For branch-scoped composer behavior, treat the composer as representing the current branch state only. Mode/config changes made from one branch’s composer must not affect other branches; child branches inherit the parent branch’s mode/config by default, and users can then switch Waggle configuration inline (for example via `/`) on that branch before sending. Do not introduce Waggle-only branch semantics; branch navigation, draft creation, materialization, and active-run behavior should mirror standard mode unless a concrete runtime limitation requires a narrow exception.
- When the user specifies an exact config-file shape, preserve that structure literally. Do not introduce an extra wrapper namespace (for example `openwaggle: {}`) unless the user explicitly asks for it.
- When asked to adapt an existing product pattern (for example t3code), preserve the interaction model and visual restraint; do not invent a louder or more ornamental style than the reference without explicit user direction.
- When a scroll bug persists after incremental patches, stop layering state on top of the existing model; remove the conflicting mechanism first and rebuild the behavior from a smaller deterministic state machine.
- For chat scroll behavior, do not combine user-message top anchoring with t3code-style stream follow. If the requested model is t3code, sending a message should stick to the bottom immediately, stream growth should follow only while the user has not opted out, and artificial spacer/anchor behavior should be removed.
- When copying t3code chat scroll behavior, port the stream-update trigger too. T3 follows the stream through running-phase timeline updates, not only through row-count changes or `ResizeObserver`.
- Stream-follow opt-out must happen on the first upward wheel/touch intent, not only after the later scroll event. Waiting for the scroll event lets an already queued auto-scroll frame fight the user and creates flicker.
- When analyzing thread-navigation regressions from data-layer migrations, keep scroll restoration out of scope except to preserve the existing product requirement; focus on instant correct-thread rendering, cache/read-model behavior, and background stream continuity.

- Scope fidelity first: when the user asks for a targeted UI change (for example remove one icon), preserve all other behavior/layout unless explicitly asked to redesign more.
- When asked to redesign a UI to match a stated visual model, do not ship superficial decoration over the old structure. Change the underlying view model/layout enough to produce the promised visual behavior, then validate with screenshots that exercise the meaningful edge case (for example a real branch/divergence), not only the linear/default case.
- Reuse existing interaction components for equivalent actions instead of creating a second visual/functionality variant. If a new surface needs “scroll to bottom,” share the existing scroll-to-bottom button and scroll helper semantics, including the same center-bottom overlay placement/visibility behavior, rather than placing a different one-off button in a toolbar.
- UI additions should follow existing component style primitives before inventing new ones. For example, Session Tree search should use the composer/card input border, radius, focus ring, and typography rather than a new pill-style guide unless the user explicitly asks for a new visual language.
- Toolbar controls should share a visual rhythm: matching heights, balanced vertical padding, harmonious spacing, and compatible corner radii. Do not place mixed-height or mismatched-radius controls in the same row unless the design intentionally calls for hierarchy.
- Tree/graph connector lines should visually terminate into their target node/dot. Avoid detached short `|_` elbows by extending connector strokes slightly underneath the node marker so the visible stroke reaches the marker edge.
- Tree search must search the persisted node data that backs the visible tree, not only hydrated renderer message objects. It should also reveal matches under collapsed ancestors by temporarily expanding search result paths without mutating persisted expansion state.
- Magic-number policy is strict when requested repository-wide: replace all inline numeric literals (including UI/layout values) with named `SCREAMING_SNAKE_CASE` constants, and keep `pnpm check:magic-numbers` green before handoff.
- DRY constant design preference: when the same numeric constant appears across files, centralize it in a shared constants module and group related constants into domain objects when that improves clarity.
- Script runtime preference: prefer TypeScript scripts (`.ts`) executed with `tsx` over `.mjs` scripts when adding or updating project scripts.
- **Never type-cast** — `as Foo`, `as unknown as Foo`, and `as Record<string, unknown>` are never acceptable. Instead, always use the proper TypeScript/Effect Schema primitives to achieve type safety from top to bottom:
  - **Union types & discriminated unions** — model data variants with `type` discriminators, not casts after runtime checks.
  - **Effect Schema validation** — decode at runtime boundaries (JSON.parse, IPC, external APIs) with the shared helpers in `src/shared/schema.ts` and shared schemas under `src/shared/schemas/`. Prefer schema-driven narrowing over permissive `unknown` fallbacks.
  - **`as const`** — use freely for literal type narrowing and tuple inference.
  - **Proper types & interfaces** — define explicit types and interfaces for all data shapes. Use explicit type annotations for mutable state (`const x: { used: boolean } = { used: false }`) instead of `false as boolean`.
  - **Generic functions** — use generics to propagate types through function boundaries instead of casting return values.
  - **Type inference top-to-bottom** — let TypeScript infer from Effect schemas (`Schema.Type<typeof schema>`), function return types, and generic parameters. If inference doesn't reach a consumer, fix the type chain instead of casting at the consumer.
  - **Type guards** — use `function isFoo(x: unknown): x is Foo` for narrowing, not casts after manual checks.
- **Never “bridge” IPC typing gaps with casts** — when Electron IPC helpers or tests hit generic friction, reshape the wrapper signatures, helper types, or test handlers so the call is typed natively. Do not use `as never`, `as unknown[]`, or handler casts to force `ipcMain.handle`/`ipcMain.on` to compile.
- **`// SAFETY:` comments are NEVER allowed** — they rationalize casts instead of fixing them. If a cast exists, the code needs a proper structural fix (type guard, generic, discriminated union, Effect Schema validation). No exceptions.
- **Use Effect Schema for runtime validation** — prefer shared schema modules plus `decodeUnknownOrThrow` / `safeDecodeUnknown` helpers over ad hoc JSON checks or permissive parse fallbacks.
- **Skill creation workflow is mandatory** — when creating or updating any skill, always load and follow the `skill-creator` skill first. Treat this as a hard requirement, not optional guidance.
- **Skill location support must be comprehensive** — when applying `skill-creator` outputs, create skills under the corresponding `skills` subfolder (`.openwaggle/skills/`, `.claude/skills/`, `.codex/skills/`) as requested by the user.
- **Never unstage/revert unknown work** — do not unstage, reset, checkout, or otherwise undo any existing unstaged/staged changes that were not explicitly implemented by this agent in the current turn, because they may belong to another agent's in-progress work.
- **Preserve user-added skills and unrelated added files** — when the user adds a skill or other files during a migration, do not delete, move, rename, stage-clean, or otherwise manage them unless explicitly asked. Keep implementation focused on the assigned work and treat unrelated added files as off-limits.
- **Avoid explicit `Record<string, unknown>` annotations when inference can express the same shape** — prefer structurally inferred objects (for example via `Object.fromEntries`) unless a named interface is truly required.
- **React component architecture preference (user-required)** — avoid pass-through wrapper components and prop-drilling-heavy composition for app-level UI. Prefer components that self-wire via focused Zustand/hooks, and keep parent components for layout composition only when they add real layout behavior.
- **Avoid mega controller props in split components** — do not pass large controller objects (or many forwarded props) through section boundaries. Decompose controller logic into focused hooks/selectors consumed directly inside each feature section.
- **Pre-user products should prefer clean-break changes over transitional scaffolding** — when the user explicitly states there are no users yet, remove inactive keys/paths directly instead of adding migration code.
- **Always acknowledge user messages** — when the user sends information (summaries, FYIs, context), always explicitly acknowledge it before or during related work. Never silently consume user-provided context without responding. Ignoring a user message is disrespectful regardless of how busy you are with a task.
- **Don't prematurely optimize** — don't add throttling, debouncing, or caching mechanisms unless there's a demonstrated performance problem. ReactMarkdown handles typical LLM streaming rates (~20-50 tokens/sec) without jank. Optimize when you measure a problem, not when you imagine one.
- **Never let Composer API drift crash the composer UI** — renderer features that depend on newly added preload methods must feature-detect runtime availability (`typeof api.method === 'function'`) and gracefully degrade with local inline errors/toasts. Do not allow a missing method to throw and trigger `Composer panel error` fallback that removes the composer interaction surface.
- **Do not render auto-converted long-prompt attachments as inline text previews in chat messages** — for generated `Pasted Text *.md` attachments, show file-only attachment labels in transcript previews. Rendering clipped content (`...`) makes users think the message was truncated/sent inline and degrades trust in attachment upload UX.
- **Do not silently swallow async failures** — if a promise rejection is intentionally absorbed for control-flow/cleanup reasons, always log structured context (component/feature + actionable error message) so failures remain diagnosable.
- **Do not add silent parse fallbacks** — when a new JSON/runtime parse path can fail, validate it with shared helpers and log structured context for failures instead of adding `catch {}` or otherwise swallowing the parse error invisibly.
- **Do not claim exhaustive verification without prompt-level evidence** — for conversational agent fixes, always run and report a concrete prompt matrix (with pass/fail outcomes) in addition to test suites before stating validation is complete.
- **Never state “100% usable/fixed” before critical-user-path live validation** — for agent runtime flows, require at least one real in-app prompt replay of the failing case (not only unit/check suites) before making absolute reliability claims.
- **Preserve continuity before context compaction** — when approaching token/credit limits, explicitly record a compact handoff summary (current hypothesis, exact failing evidence, files touched, next commands) so no critical debugging context is lost across compaction.
- **Keep a rolling handoff during long autonomous runs** — don’t wait until the very end of context; refresh the compact handoff after each major phase (repro, fix, verification) so compaction can happen safely at any point.
- **Use the `apply_patch` tool directly for patches** — never invoke `apply_patch` through `exec_command`; direct tool usage avoids command-wrapper failures and preserves deterministic patch semantics.
- When a user questions whether a new library integration follows best practices, do a second-pass review against the library’s official docs and tighten any merely-functional patterns before handoff; don’t stop at “tests pass” if the abstraction is still being used awkwardly.
- When the user wants a reusable repository-wide integration guide for a major dependency, create a broad project skill (for example `pi-integration`) that captures both gotchas and important documented usage patterns, including where the official docs/examples live and how to use the SDK in this repository.
- When implementation work outgrows a planning/docs branch, move it onto a representative feature branch before continuing. Do not keep substantial product implementation on a docs-named branch.
- **Compact composer interactions must stay compact** — when the user asks for a Codex-style inline control inside the composer, do not expand it into a separate panel or review surface. Keep the interaction in-row unless the user explicitly asks for a larger multi-step UX. Branch-summary choices for draft branch creation must be composer-integrated rather than separate popovers/modals; custom summary instructions reuse the composer input surface but route to Pi branch summarization, not normal chat send. Never destroy user intent already typed in the composer; temporary composer modes must preserve and restore existing unsent prompts/drafts when canceled or when the user navigates away. Composer drafts must be scoped to the concrete session/branch/draft context, not global app state. The visible composer value should be left alone unless the user navigates to a different project/session/branch context; on navigation, save the current context's draft and load the target context's draft so prompts do not leak between contexts. Lifecycle actions like archive/restore/delete should not directly mutate the visible composer except through any resulting navigation. Archived conversations/sessions/branches do not retain stored composer drafts for later restore; restoring archived items restores conversation history, not old unsent composer drafts.
- **Match the user’s placement sketch literally for composer micro-UX** — if the user marks a specific toolbar region for a control, preserve the textarea/body height and swap content only within that marked row instead of relocating the interaction elsewhere in the composer.
- **After migration regressions, run a real E2E check before claiming the path is fixed** — focused unit/component/build checks are not enough for click-through UX flows in Electron. Validate the live user path at least once in headless E2E before handoff.
- **Logging is performance-sensitive infrastructure** — never log in hot paths (stream chunks, tight loops, high-frequency IPC/render cycles), never eagerly stringify/build log payloads, keep debug tracing opt-in only, and for retry/polling flows log only the first retry and the terminal outcome.
- **Unit, integration, and component tests belong in nearby `__tests__/` folders** — when adding or moving non-E2E tests, colocate them in a local `__tests__/` directory next to the feature area instead of leaving `*.test.*` files loose beside implementation files. E2E remains under `e2e/`.
- **Do not create spec markdown files for straightforward cleanup passes unless the user asks for one** — repository hygiene tasks like file moves or test colocation should be executed directly without adding unnecessary planning documents.
- **Fix transcript/steer regressions from the actual async boundary, not from UI intuition** — when a queued or steered user message is missing, trace the exact composer → queue → steer/send IPC ordering first and patch the earliest boundary that can delay visibility. Do not keep changing downstream transcript rendering until the real ordering bug is proven.
- **Disambiguate “terminal” requests before changing the renderer terminal UI** — when the user mentions terminal output during install or release flows, confirm whether they mean the shell script output versus the in-app terminal panel, and implement on the named surface only.
- **Never use inline `import()` type syntax in source files** — always import types at the top of the file. Inline `import('@shared/types/foo').Bar` hides dependencies, breaks refactoring tools, and violates the project's explicit import convention. The only legitimate use is in `.d.ts` files or to break circular dependencies.
- **Use `while (true)` for infinite loops, never `for (;;)`** — `while (true)` reads as plain English and is immediately understood. `for (;;)` is a C/systems idiom that is less common in JS/TS and creates unnecessary readability friction.
- **After a context transition, continue the actual task** — if the user says keep going after a handoff or summary, continue the work instead of pausing on process commentary.
- **Keep Pi-native transport semantics end-to-end** — when transport/runtime work touches Pi session events, keep Pi-native event semantics as the source of truth. Do not translate Pi events into a separate stream protocol for convenience; if the flow becomes hard to follow, stop and redesign around Pi-native transport instead of layering another mapper.
- **Provider metadata must mirror Pi TUI one-to-one** — use the same provider/model/auth implementation model Pi TUI uses: Pi `AuthStorage`, `ModelRegistry`, built-in provider support, and Pi-native custom-provider/custom-model extension points. OpenWaggle should add UI around Pi’s auth methods (OAuth, API key, other Pi-supported flows) while keeping runtime metadata sourced from Pi.
- **Pi model identity includes its provider** — every Pi-hosted model variant must be treated as a distinct option using `provider/modelId`, even when several providers expose the same model id (for example `gpt-5.4`). Provider logos and display names are UI polish only; auth, selection, persistence, and runtime routing must follow Pi’s provider/model semantics exactly.
- **OpenWaggle is a UI on top of Pi, not a custom runtime/tool layer** — start from Pi’s native tools/events/session behavior and build UI support for what Pi provides. Runtime/tool features should enter through explicit Pi adapter boundaries.
- **Pi-native sidebar navigation is Projects-only** — do not add a separate global “Chats” section or projectless chat fallback. Sessions belong under project sections, and project switching should start/select project context instead of mutating an existing Pi session’s `projectPath`.
- **Pi provider auth UI is method-based** — Settings should group provider auth by Pi-supported method (API key, OAuth, and future Pi-native auth methods), keep those groups collapsed by default, and start OAuth through Pi’s browser flow. OAuth rows should use the compact toggle interaction, and that toggle must remain clickable during login so users can cancel an in-progress auth flow.
- **Do not use provider-level auth state for method-specific rows** — a provider can be available through OAuth while its API-key method is unconfigured. Keep provider-level runtime availability separate from API-key configured state and OAuth connected state so Settings never makes one auth method appear active because another method is connected.
- **Manual context compaction must mirror Pi TUI slash-command UX** — the context meter is informational only. Do not trigger manual compaction from the meter. Support `/compact` and `/compact <custom instructions>` through the composer slash-command flow, passing custom text to Pi as compaction instructions.
- **Do not over-protect local dev processes after clarification** — if the user clarifies that a running app/dev process is theirs and gives permission to restart or attach, proceed with the required validation instead of continuing to defer live QA.
- When a user asks to install/configure Pi itself, treat it as user-level Pi configuration unless they explicitly say project-local. Do not modify the current repository or project `.pi/settings.json` for Pi CLI experiments, and prefer user-level opt-in aliases for large extensions.
- **Do not add legacy compatibility infrastructure during Pi-native cleanup** — when Codex/review feedback asks for backfills or dual reads from pre-Pi OpenWaggle stores, prefer removing stale legacy paths or making the current implementation explicitly Pi-native. Do not preserve old `conversations`, `auth_tokens`, provider settings, or transitional shims unless the user explicitly asks for an upgrade compatibility layer.
- **OpenWaggle skills must be runtime-supported, not only cataloged** — `.openwaggle/skills` is an OpenWaggle product location and should be bridged into Pi alongside `.pi/skills` and `.agents/skills`, with catalog toggles respected where OpenWaggle owns the catalog.
- **Project config is `.openwaggle/settings.json`** — preserve OpenWaggle top-level keys and keep Pi runtime settings nested under `pi`.
- **Do not track user-specific settings files** — project-local and global/user-level `settings.json` files (for example `.openwaggle/settings.json` and `.pi/settings.json`) are per-user runtime configuration and should be ignored for every clone. If shared defaults are needed, commit a clearly non-secret template/default file instead and keep real settings untracked.
