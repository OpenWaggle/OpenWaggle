# Agent Issue Tracker Context

OpenWaggle uses GitHub Issues in `OpenWaggle/OpenWaggle` and tracks roadmap work in the GitHub organization project referenced from `CONTRIBUTING.md`.

## Tracker

- Repository: `OpenWaggle/OpenWaggle`
- Issues: `https://github.com/OpenWaggle/OpenWaggle/issues`
- Roadmap: `https://github.com/orgs/OpenWaggle/projects/1`
- Discussions: `https://github.com/OpenWaggle/OpenWaggle/discussions`

## Contribution Model

The public contribution model currently prioritizes issues, bug reports, feature requests, documentation feedback, UI/UX feedback, and architecture proposals. Public code contributions are not generally open yet, so issue quality matters.

For internal agent work, use the repository branch/PR conventions in `AGENTS.md` and preserve the no-commit-without-maintainer-approval rule.

## Issue Writing Rules

Use the product vocabulary from `docs/agents/domain.md`.

Good issues:

- describe the user-visible behavior or product outcome
- include concrete acceptance criteria
- avoid stale implementation paths unless needed to point at a known seam
- note relevant first principles or architecture constraints
- separate product decisions from implementation tasks

For bugs, include:

- observed behavior
- expected behavior
- reproduction steps or captured evidence
- environment details when relevant
- logs, screenshots, or failing commands when available

For implementation slices, prefer vertical tracer bullets over horizontal layer tasks.

## Matt Skill Expectations

### `to-prd`

Create a GitHub issue containing the PRD. Apply `ready-for-agent` when the PRD is sufficiently specified for agent implementation.

### `to-issues`

Publish approved vertical slices as GitHub issues in dependency order. Use `ready-for-agent` for independently grabbable AFK work. Use `ready-for-human` when the slice requires maintainer judgment, external access, design approval, or manual release/distribution work.

### `triage`

Move issues through the triage labels documented in `docs/agents/triage-labels.md`. Any issue comment posted by the triage skill must start with:

```markdown
> *This was generated by AI during triage.*
```

## Pull Requests

Do not create commits, push, or open PRs unless the maintainer explicitly approves. When a PR is approved for creation:

- link solving work with `Closes #<issue-number>`
- link partial progress with `Part of #<issue-number>`
- include validation evidence
- include release notes for user-visible changes until release-intent files exist
